# TECNOPRO — Auditoría de UX/UI

**Fecha**: 2026-07-02
**Contexto**: Post-MVP (Fase 2 completa + Fase 3.1 mergeada). Antes de continuar con Fase 3.2/3.3, revisión completa del sistema para detectar issues de usabilidad y lógica de negocio.
**Motivación**: feedback del owner del proyecto — el sistema se construyó siguiendo un scope de 14 módulos pero sin suficiente reflexión sobre la experiencia real de un dueño de service técnico usándolo todos los días.

---

## Resumen ejecutivo

**Total**: 32 issues detectados.

| Severidad | Cantidad | Descripción |
|---|---|---|
| **🔴 Crítico** | 5 | Rompen usabilidad diaria o generan errores de negocio graves |
| **🟠 Alto** | 8 | Fricción constante, workarounds necesarios |
| **🟡 Medio** | 14 | Mejoras significativas de experiencia |
| **🟢 Bajo** | 5 | Pulido, cosmética, detalles |

**Área más afectada**: módulo **Plata** (Caja + Gastos + Tesorería + Contabilidad) — arquitectura fragmentada, labels técnicos, faltan flujos completos.

**Fixes con mayor ROI** (impacto/esfuerzo):
1. Sistema de inputs numéricos (afecta 11 forms)
2. CRUD de categorías (repuestos + gastos + ubicaciones)
3. Stock inicial en `/stock/nuevo`
4. Rediseño de módulos Plata (Caja absorbe Gastos, Contabilidad absorbe Tesorería)
5. Sistema de toasts (reemplaza 15 usos de `alert`/`confirm` nativos)

## 🔴 Issues críticos

### C1. Inputs numéricos sin formato de miles

**Files afectados**: 11 forms
- `components/servicios/ServicioForm.tsx` (precio_base)
- `components/repuestos/RepuestoForm.tsx` (costo, precio_venta, stock_minimo)
- `components/repuestos/MovimientoStockForm.tsx` (cantidad)
- `components/ordenes/AgregarServicioForm.tsx`, `AgregarRepuestoForm.tsx` (precio, cantidad)
- `components/presupuestos/PresupuestoForm.tsx` (margen_pct)
- `components/presupuestos/AgregarServicioPresForm.tsx`, `AgregarRepuestoPresForm.tsx`
- `components/caja/MovimientoForm.tsx`, `CobrarOrdenForm.tsx` (monto)
- `components/gastos/GastoForm.tsx` (monto)

**Problema**: `<Input type="number" />` muestra `100000` en vez de `100.000`. Guillo puede escribir `10000000` pensando que son $10 millones cuando quiso $1 millón. **Es un sistema que maneja plata sin formato de plata**.

**Fix propuesto**: componente `MoneyInput` (o `NumberInput` con prop `thousandSeparator`) que:
- Muestra `100.000` con separadores mientras se escribe.
- Guarda el número pelado en state (100000).
- Placeholder `0` en vez de valor inicial 0 pegado (ver C2).
- Prefijo opcional `$` para inputs de plata (visible).

**Alcance**: crear componente + reemplazar en los 11 forms + tests manuales.

### C2. Inputs numéricos con "0" pegado al valor

**Files afectados**: mismos 11 forms + algunos con `value={form.x}` en tipo number.

**Problema**: `value={form.costo}` cuando `form.costo = 0` renderiza el `"0"` en el input. Al escribir `25`, el DOM queda `025` o `250` según posición del cursor — el user tiene que borrar el 0 primero.

**Fix**: `value={form.field || ""}` con `placeholder="0"`. El 0 aparece como placeholder gris pero desaparece al escribir. Ya lo hago en `/caja/nuevo` y `/gastos/nuevo`; falta en los otros.

**Alcance**: 8-9 forms más para actualizar.

### C3. Categorías de repuestos como texto libre (invita typos)

**File**: `components/repuestos/RepuestoForm.tsx:85-92`

```tsx
<Input
  id="categoria"
  value={form.categoria ?? ""}
  onChange={(e) => update("categoria", e.target.value)}
  placeholder="Ej. Memorias"
/>
```

**Problema**: Guillo carga "Memoria RAM" hoy, la semana que viene "Memorias RAM", después "MEMORIA" — se van creando categorías fantasma por typo. La lista de "categorías" se ensucia y los filtros dejan de servir.

**Fix propuesto**: componente `CategoriaComboBox`:
- Lee `SELECT DISTINCT categoria FROM repuestos WHERE categoria IS NOT NULL AND activo = true`.
- Muestra las existentes en un dropdown.
- Permite escribir libre — si el texto no matchea ninguna, ofrece "Crear nueva categoría".
- Al submit, si es nueva, se persiste con el repuesto (sin tabla separada — sigue viviendo en la columna `categoria`).

**Mismo pattern aplica a**: `ubicacion` (Ubicación en depósito, `RepuestoForm.tsx:151-158`).

**Nota**: para las categorías de repuestos, dado que hoy están como columna en `repuestos`, se puede mantener así con este UX. Si en el futuro se quiere convertirlas en tabla separada (para editar/renombrar), es una migración chica.

### C4. Categorías de gasto sin CRUD desde UI

**Files afectados**: `/configuracion` sin sección de categorías; `/gastos/nuevo` solo permite elegir de una lista pre-seedeada.

**Problema**: en Ola C.2 (Gastos) elegimos "categorías configurables en tabla, no enum" — y el owner lo aprobó pensando que después se podía gestionar desde UI. **Pero no se implementó la UI**. Hoy Guillo tiene los 8 seeds y para agregar/desactivar/renombrar una categoría **tiene que ir a Supabase SQL Editor**. Esto rompe todo el argumento de "configurable".

**Fix propuesto**: página `/configuracion/categorias-gasto` con:
- Lista de categorías (nombre, orden, activa/inactiva).
- Botón "Nueva categoría".
- Edición inline (renombrar).
- Toggle activo/inactivo.
- Reordenar drag-and-drop (o botones ↑ ↓ para MVP).
- Al desactivar una categoría con gastos históricos, mantenerla como opción de solo lectura (no borrar).

**Mismo pattern aplica a**: umbrales de configuración (ver M4/M10) — cualquier cosa "configurable" necesita su UI.

### C5. Stock inicial: flow de 2 pasos con caja confusa

**Files afectados**: `RepuestoForm.tsx:161-165`, ficha `/stock/[id]`.

```tsx
{mode === "create" && (
  <p className="text-xs font-mono text-tp-muted">
    El stock inicial es 0. Cargalo desde la ficha del repuesto con un movimiento tipo ENTRADA.
  </p>
)}
```

**Problema**: Guillo va a arrancar con **cientos de repuestos ya en el depósito** — todos comprados hace tiempo. Cargar un repuesto con stock 15 hoy requiere:
1. Crear el repuesto (stock=0).
2. Ir a la ficha.
3. Cargar movimiento ENTRADA con cantidad=15.

Además, la duda razonable: **"¿esto no me va a generar un egreso en caja?"**. No lo hace (los movimientos de repuestos NO tocan caja), pero la falta de un mensaje explícito genera fricción y potencialmente errores contables ("me anoté que gasté $50.000 en RAM pero después Guillo cargó un gasto pensando que era otra cosa").

**Fix propuesto**: en `/stock/nuevo` agregar sección al fondo:

```
─── Stock inicial (opcional) ────────────────────
[ ] Ya tengo unidades de este repuesto en depósito

  Si marcás esto, se cargan estas unidades como stock inicial.
  NO afecta la caja — asumimos que las tenías desde antes.

  Cantidad inicial: [___]
  Motivo: [Stock inicial de carga]  (autocompletado, editable)
```

**Lógica**: al crear el repuesto, si `cantidad_inicial > 0`, insertar automáticamente un movimiento ENTRADA con motivo "Stock inicial de carga". Este movimiento NO toca caja (nunca lo hizo). El texto explícito "no afecta caja" elimina la duda.

**Alternativa**: si el user quiere reflejar la compra histórica en caja, tiene la opción de cargar un gasto manual (que ya existe). Pero por default, stock inicial ≠ egreso.

## 🟠 Issues altos

### A1. Labels técnicos "Tipo/Origen" en `/caja/nuevo`

**File**: `components/caja/MovimientoForm.tsx`

**Problema**: los labels vienen literalmente del schema SQL — "Tipo" (INGRESO/EGRESO) y "Origen" (APERTURA/CIERRE/AJUSTE/OTRO). El screenshot que mostró el owner es autoexplicativo: **no se entiende qué es cada cosa**.

**Fix**:
- `Tipo` → **"¿Entra o sale plata?"** con opciones `Entra` / `Sale`.
- `Origen` → **"Motivo"** con opciones más legibles (`Apertura de caja`, `Cierre de caja`, `Ajuste de conteo`, `Otro`).
- Helper text: *"Si es un gasto categorizable (proveedor, luz, alquiler), cargalo en el módulo Gastos — se registra automáticamente acá."*

**Alcance**: este fix se hace junto con el rediseño Plata (ver sección aparte más abajo).

### A2. Tiempo estimado del servicio solo en minutos

**File**: `components/servicios/ServicioForm.tsx:102-113`

**Problema**: cargar "8 horas" implica escribir `480`. Cargar "3 días" implica calcular minutos (¿1440 por día? ¿solo horas hábiles?). Absurdo.

**Fix**: dos inputs — cantidad + unidad (`minutos` / `horas` / `días`). Almacenamos en minutos en DB (columna sigue igual, `tiempo_estimado_min`) pero calculamos en cliente al mostrar. Ejemplo:

```tsx
<div className="grid grid-cols-2 gap-2">
  <Input type="number" value={form.tiempo_valor} onChange={...} />
  <Select value={form.tiempo_unidad} onChange={...}>
    <option value="min">Minutos</option>
    <option value="h">Horas</option>
    <option value="d">Días</option>
  </Select>
</div>
```

Guardar: `tiempo_estimado_min = valor * factor(unidad)`. Al leer para editar, hacer el reverso.

### A3. Código interno de repuesto no autogenerado si el user lo deja vacío

**File**: `components/repuestos/RepuestoForm.tsx:74-82`

**Problema**: si Guillo no carga código interno, queda `null`. En la lista, la columna código queda vacía. Menos legible.

**Fix**: si `codigo` es null/vacío al guardar, copiar `id_publico` (REP-XXXX). Editable después manualmente. Ya se conversó, confirmado.

### A4. Rediseño arquitectural: módulos Plata con overlap

**Files afectados**: `/caja`, `/gastos`, `/tesoreria`, `/contabilidad`.

**Problema**: 4 módulos que se solapan conceptualmente. Guillo tiene que decidir "¿esto va en Caja o en Gastos?" cada vez.

**Decisión del owner** (confirmada): **unificar**.

**Rediseño propuesto**:

#### `/caja`
- **Todos los movimientos**, con filtros ricos (por tipo, motivo, categoría, fecha, método de pago).
- Botón principal "Registrar movimiento" con **wizard/tabs internos**:
  - Tab 1: **Ingreso rápido** (apertura, ajuste, otro ingreso ad-hoc).
  - Tab 2: **Egreso categorizado** (equivalente al actual "Nuevo gasto" — con categoría, descripción, notas). Detrás usa el RPC `registrar_gasto`.
  - Tab 3: **Cierre de caja** (opcional, flow guiado).
- KPIs de saldo actual + acumulados del mes.
- Los cobros de órdenes siguen entrando desde la ficha de orden como hoy (no cambia).

#### `/contabilidad`
- Libro completo (lo que hoy tiene).
- **Pestaña "Por cobrar"** (lo que hoy es `/tesoreria`): órdenes con saldo pendiente, resumen mensual ingresos vs egresos.
- **Pestaña "Libro"** (lo que hoy es contabilidad): filtros por período + export CSV.

#### Eliminar del nav:
- `/gastos` (se accede desde Caja tab "Egreso categorizado").
- `/tesoreria` (se accede desde Contabilidad pestaña "Por cobrar").

#### Impacto:
- Nav: de 14 items a 12.
- Rutas `/gastos` y `/tesoreria` quedan como redirects a `/caja?tab=egreso` y `/contabilidad?tab=por-cobrar` (para no romper bookmarks/history).
- Migración cero — solo UI.
- Backend intacto (tablas y RPCs siguen igual).

**Alcance**: mediano. 1 PR grande.

### A5. `alert()` y `window.confirm()` nativos por toda la app

**Files afectados**: 15 componentes (grep confirma).

**Problema**: los diálogos nativos del browser son feos, no se pueden estilar, cortan el flow. En una app profesional dan sensación de amateur.

**Fix**: sistema de **toasts** (para mensajes) + **confirm dialog custom** (para confirmaciones destructivas):
- Usar Radix Dialog (ya está instalado) para el confirm.
- Toast simple con `react-hot-toast` o custom con Radix — evaluar según peso.

Componentes a migrar:
1. Todos los `alert(result.error)` → `toast.error(result.error)`.
2. Todos los `window.confirm(...)` → `<ConfirmDialog>`.
3. Agregar `toast.success(...)` en actions que hoy hacen redirect silencioso (ver A8).

**Alcance**: 15 archivos + 2 componentes nuevos (Toast + ConfirmDialog).

**File**: `app/(auth)/login/`

### A8. Falta feedback tras acciones exitosas (redirect silencioso)

**Files afectados**: la mayoría de actions que hacen `redirect(...)` al final.

**Problema**: creás un cliente → click Guardar → aparecés en la ficha sin mensaje de "se creó". Peor: creás una orden → aparecés en la lista, sin saber si se creó (a menos que la busques).

**Fix**: sistema de toasts con mensajes tipo "Cliente CLI-0042 creado". Ya está mencionado en A5, es el otro pilar del pattern.

## 🟡 Issues medios

### M1. Filas no clickeables en varias listas
Aplicado en `/ordenes` y `/presupuestos` (con estado inline). **Falta** en: `/clientes`, `/catalogo`, `/stock`, `/usuarios`, `/turnos` (vista lista). Pattern documentado en `docs/PATRONES.md`.

### M2. Panel: KPIs sin comparativa temporal
Hoy muestra "saldo actual" pero no "vs ayer" o "vs mes pasado". Sin tendencia. Agregar mini indicador `+15%` o similar.

### M3. Analytics: sin filtro por período
Hoy los charts son "todo lo que hay" o "mes actual" fijo. Agregar preset mes/año/custom (como Contabilidad).

### M4. Alertas: umbrales hardcoded en el código
`DIAS_SALDO_VENCIDO = 30`, `DIAS_PRESUPUESTO_PROXIMO = 7`. Deberían estar en tabla `configuracion` y editables desde `/configuracion`. Fix junto con C4 y M10.

### M5. Sin loading skeletons
Todas las páginas usan server components con `await` — el usuario ve loader del browser. Agregar `<Suspense>` con skeletons personalizados en las listas y en el panel.

### M6. Formato de fecha inconsistente
En algunas tablas se ve `27/6/2026`, en otras `27/06/2026`, en otras `27 jun 2026`. Auditar y unificar con `formatFecha`.

### M7. Sin mobile responsive de tablas grandes
Tablas con 6-7 columnas rompen en mobile. Fix: en mobile, cada fila como card con las columnas apiladas.

### M8. Sin búsqueda global (cmd+K)
Buscar "OT-0042" hoy requiere ir a `/ordenes` primero. Un cmd+K global que busque en órdenes, presupuestos, clientes y muestre resultados agrupados ahorra mucho tiempo.

### M9. Historial no expuesto en UI
Tabla `historial` se llena con todas las acciones importantes pero solo se ve desde SQL. Agregar `/historial` (admin-only) con filtros.

### M10. Configuración: sección de umbrales de alertas
Agregar en `/configuracion` los 4 umbrales (días saldo, días presupuesto, stock alerta días, margen default). Ya existen 3 en la tabla `configuracion`, faltan agregar los de alertas.

### M11. Notas internas de orden no destacadas
En `/ordenes/[id]`, la sección "Notas internas" se ve igual que el resto — poco destacada. Agregar borde/color diferenciado (patrón "post-it" o similar).

### M12. Turnos: fecha_fin no sugerida
Al cargar un turno nuevo, `fecha_fin` está vacía. Default sugerido = `fecha_inicio + 1h`, editable.

### M13. Confirmación de guardado en items imputados
Al agregar servicio/repuesto a una orden, no hay confirmación visible — solo se actualiza la lista. Toast "Servicio agregado" mejora la sensación.

### M14. `/presupuestos/[id]` sin timeline visual
Estados BORRADOR → ENVIADO → APROBADO son un flow. Mostrar como stepper visual ayuda a Guillo a saber en qué punto está.

## 🟢 Issues bajos

### B1. Tokens `tp-*` sin documentación visual
Un Storybook o página `/design-system` interna que muestre paletas, tipografía, componentes.

### B2. Iconos hardcoded en algunos places
Algunos componentes importan íconos directamente en vez de usar el mapping. Sin impacto real, es limpieza.

### B3. Empty states inconsistentes en tono
Algunos dicen "Todavía no hay X", otros "No hay X registrado", otros con ejemplo. Unificar el tono.

Verificar que no se indexe públicamente (aunque require auth, el HTML del login sí es público).

## Rediseño de módulos Plata (propuesta detallada)

### Situación actual

| Ruta | Contenido | Overlap |
|---|---|---|
| `/caja` | Todos los movimientos + saldo | Base — no overlap |
| `/gastos` | Egresos categorizados | Se solapa con `/caja` (todo gasto es un movimiento) |
| `/tesoreria` | Órdenes con saldo pendiente + resumen mensual | Se solapa con `/contabilidad` (mismos movimientos, otro ángulo) y con Panel (KPI "por cobrar" repetido) |
| `/contabilidad` | Libro completo + export CSV | Base — no overlap |

### Situación propuesta

| Ruta | Contenido | Sub-secciones |
|---|---|---|
| `/caja` | Todos los movimientos + saldo | Tabs de "Nuevo movimiento": Ingreso rápido / Egreso categorizado (ex-gastos) / Cierre |
| `/contabilidad` | Libro + Por cobrar | Pestañas: Libro (ex-actual) / Por cobrar (ex-tesorería) |

**Ítems del nav antes**: 14 → **después**: 12.

**Grupos del nav actualizados**:

```
Plata:
- Caja
- Contabilidad

Análisis:
- Analytics
- Alertas
```

### Migración

- Sin cambios SQL — tablas y RPCs quedan igual.
- `/gastos` → redirect 301 a `/caja?nuevo=gasto`.
- `/tesoreria` → redirect 301 a `/contabilidad?pestaña=por-cobrar`.
- UI: mover `GastoForm` como tab dentro de `MovimientoForm`; mover contenido de `/tesoreria` a `/contabilidad` como pestaña.
- Nav: quitar 2 items, ajustar orden.
- Panel: actualizar los links de las KPIs (por cobrar apunta a Contabilidad).

**PR estimado**: 1 grande. Se puede partir en 2 (Caja+Gastos primero, Contabilidad+Tesorería después).

- ✅ Cada acción exitosa muestra confirmación visual (toast).
- ✅ Cero `alert()` nativos del browser en el código.
- ✅ Nav tiene 12 items en vez de 14 (Plata simplificado).
